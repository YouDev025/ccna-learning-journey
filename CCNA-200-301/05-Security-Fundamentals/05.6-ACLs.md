# ACLs (Access Control Lists)

## Definition
ACLs are ordered rules used to permit or deny traffic based on criteria such as source address, destination address, protocol, or port.

## Simple explanation
An ACL works like a checklist. Traffic is compared from top to bottom, and the first matching rule decides what happens.

## ACL types
| Type | Filters by | Common placement |
|---|---|---|
| Standard ACL | Source IP only | Close to destination |
| Extended ACL | Source, destination, protocol, port | Close to source |

## Important ACL rules
- ACLs are processed top to bottom.
- First match wins.
- There is an implicit `deny any` at the end.
- More specific entries should usually be placed before broad entries.
- ACLs can be applied inbound or outbound on an interface.

## Standard ACL example
```text
access-list 10 permit 192.168.10.0 0.0.0.255
interface gigabitEthernet0/1
 ip access-group 10 in
```

## Extended ACL example
```text
access-list 100 deny tcp 192.168.30.0 0.0.0.255 host 192.168.10.10 eq 80
access-list 100 permit ip any any
interface gigabitEthernet0/0
 ip access-group 100 in
```

## Named ACL example
```text
ip access-list extended BLOCK_GUEST_HTTP
 deny tcp 192.168.30.0 0.0.0.255 host 192.168.10.10 eq 80
 permit ip any any
```

## Wildcard mask reminder
A wildcard mask tells IOS which bits to ignore. `0.0.0.255` matches any host in a `/24` network.

## Useful commands
```text
show access-lists
show ip interface
show running-config | section access-list
```

## Common mistakes
- Forgetting the implicit deny.
- Applying the ACL in the wrong direction.
- Placing entries in the wrong order.
- Using the wrong wildcard mask.
- Blocking return traffic accidentally.

## Exam focus
Standard ACLs filter mainly by source IP. Extended ACLs can filter by source, destination, protocol, and port. ACL order matters.

## Quick review
ACLs control traffic by matching ordered permit and deny statements.
